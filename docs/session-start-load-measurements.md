# Session Start Load Measurements

This document records measured evidence for issue
[#105](https://github.com/ssheld/agent-vault/issues/105). It intentionally
separates startup context, protocol reads, and behavioral recall so follow-up
changes do not assume more than the data shows.

## Evidence Types

| Evidence type | Meaning | Current source |
| --- | --- | --- |
| Startup context | File content is present before the agent takes its first action. | Codex: `codex debug prompt-input`. Claude: no direct request-body inspection available from the tested CLI debug paths. |
| Protocol read | The agent opened or searched a file after session start while following instructions. | Claude: JSONL tool-call traces parsed by `scripts/parse-claude-memory-trace.py`. Subagent internal tool calls are not recorded in the main JSONL in current Claude Code; subagent rows reflect Agent invocation only. Codex: `codex exec --json` event stream command executions. |
| Behavioral recall | The agent follows a sentinel instruction without the evaluator naming the source file. | Prompt templates generated by `scripts/measure-agent-memory-load.sh`; Claude tool-disabled recall probe recorded below. |
| Intended import graph | A wrapper or import directive references a file that should be loaded by the host. | Manual resolver or static inspection; this is not proof that the host transmitted the content. |

## Measurement Helper

Run:

```sh
bash scripts/measure-agent-memory-load.sh
```

The helper creates a temporary generated repository, injects unique sentinels
into canonical `agent-vault` memory files, and writes:

- `sentinel-manifest.tsv` with file path, label, sentinel, behavioral answer,
  byte count, and line count
- `codex-startup-context.tsv` with Codex startup-context results
- `prompts/protocol-following-prompt.md`
- `prompts/behavioral-recall-prompt.md`
- raw Codex prompt-input JSON outputs when Codex measurement runs

Use `--skip-codex` when only fixture generation and prompt templates are needed.
Use `--output-dir <path>` when the fixture must live at a stable path for manual
agent runs.

## Claude Inspectability Spike

Claude's spike result is recorded in issue
[#105](https://github.com/ssheld/agent-vault/issues/105#issuecomment-4367130105).

Summary:

- `claude --debug api --debug-file ...` did not expose resolved `CLAUDE.md`
  import content.
- `--debug` without a category, `--output-format json`,
  `--output-format stream-json`, and `ANTHROPIC_LOG=debug` also did not expose
  the request body content needed to grep for sentinels.
- `claude --bare` could not be used as a clean negative control for an OAuth
  session because it requires `ANTHROPIC_API_KEY`.
- Token-count deltas did provide an indirect loading signal in the spike:
  `cache_creation_input_tokens` increased from `17153` in the control cwd to
  `17540` in the fixture cwd, with `cache_read_input_tokens` unchanged at
  `16319`.

Interpretation: Claude measurement should not rely on built-in request-body
debug logs for startup-context identity. Subsequent measurements should pair:

- a manual `@` import resolver for intended content
- cache-creation-token deltas for indirect proof that additional content was
  loaded
- JSONL tool traces for protocol reads, split by `isSidechain`
- behavioral recall probes with read/search tools disabled where practical

## Claude JSONL Parser

`scripts/parse-claude-memory-trace.py` consumes a Claude Code session JSONL and
the Slice 1 sentinel manifest, then emits a TSV with one row per
`(agent_type, subagent_type, session_phase, file)` cell observed in the trace.
The matrix is dense across manifest files: every file gets a row in every
observed cell, with `tool_read=false` and `read_confidence=none` when no
evidence was found.

### CLI

```sh
python3 scripts/parse-claude-memory-trace.py \
  --jsonl <session.jsonl> \
  --manifest <sentinel-manifest.tsv> \
  [--output <out.tsv>]
```

A session-id ergonomic form is also supported when the helper can resolve the
JSONL under the standard Claude Code directory:

```sh
python3 scripts/parse-claude-memory-trace.py \
  --session-id <session-uuid> \
  --cwd <fixture-or-project-cwd> \
  --manifest <sentinel-manifest.tsv>
```

The session id resolves to
`~/.claude/projects/<encoded-cwd>/<session-id>.jsonl`, where `<encoded-cwd>` is
the absolute cwd encoded into Claude Code's project-directory form. Current
Claude Code paths replace `/` and punctuation such as `_` and `.` with `-`; the
parser tries both the older slash-only form and the punctuation-normalized form.
If resolution fails the parser errors out with a clear message rather than
guessing.

### Output Columns

`agent`, `agent_type`, `subagent_type`, `session_phase`, `cwd`, `file`,
`tool_read`, `read_confidence`, `read_evidence`, `first_read_event_index`,
`first_read_uuid`, `notes`.

### Confidence Levels

| `read_confidence` | `tool_read` | Meaning |
| --- | --- | --- |
| `high` | `true` | `Read` tool with exact path, `Grep` scoped to the file or its directory, or a Bash file-read command (`cat`, `head`, `tail`, `less`, `more`, `bat`, `nl`) targeting the file. |
| `medium` | `true` | `Grep` pattern mentioning the file path, basename-only `Read` match, or Bash search/transform commands (`grep`, `rg`, `awk`, `sed`, `sort`, `wc`, etc.) targeting the file. |
| `low_path_mention_only` | `false` | Path appears in command text but command shape does not prove the content was read. Recorded for reviewer inspection without inflating protocol compliance. |
| `none` | `false` | No evidence for this file in this cell. |

### Path Normalization

Tool inputs that use absolute paths under the entry's `cwd` are normalized to
repo-relative form before comparison against the manifest. `Grep` scoped to
`/fixture/agent-vault/` from a cwd of `/fixture` matches the manifest entry
`agent-vault/context-log.md` as `high`. Leading `./` prefixes on relative
paths are stripped (so `./agent-vault/` matches `agent-vault/...`), and
`Grep` with `path="."` or `path="./"` is treated as a broad cwd-scoped
search that covers every manifest entry. Bash commands that begin with
`cd <target> && <rest>` are parsed: the cd target becomes the effective cwd
for classifying the rest of the command, so `cd /fixture && cat agent-vault/...`
records as `high` even when the entry's recorded cwd is unrelated.

### First-Read vs. Strongest-Confidence Separation

`first_read_event_index` and `first_read_uuid` always point to the earliest
read of the file in the cell, regardless of confidence. `read_confidence` and
`read_evidence` always reflect the strongest evidence found in the cell, with
ties broken by earliest event. So an early `wc <file>` followed by a later
`Read` of the same file produces a row with `read_confidence=high` (from the
later Read) but `first_read_event_index` pointing to the earlier `wc` event.

### `/clear` Boundary Detection

The parser detects user-role JSONL entries whose text contains the literal
`<command-name>/clear</command-name>` and marks events before that boundary as
`fresh_start` and after as `post_clear`. If no marker appears, all events are
classified `fresh_start`. Variants such as `/cleared` or partial matches do not
trigger detection.

### Subagent Cell Emission

Subagent rows are emitted only for `(subagent_type, session_phase)` pairs that
were actually observed: an Agent invocation (or a future `isSidechain: true`
record, for forward compatibility) must have happened in that phase. A
subagent invoked only before `/clear` does not get a fabricated `post_clear`
row, and vice versa. Main-agent cells follow phases actually traversed in the
trace: `fresh_start` always, `post_clear` only when a `/clear` marker was
observed.

### Subagent Visibility Gap

Verified empirically across 44 local Claude Code sessions: no entries had
`isSidechain: true`. The Agent tool's invocation is recorded in the main JSONL
(with `subagent_type` in its `input`), and the Agent's `tool_result` carries
the subagent's final synthesized response, but the subagent's internal tool
calls are not stored in the main JSONL.

The parser therefore emits subagent rows only when an Agent tool invocation is
observed, with `tool_read=false`, `read_confidence=none`, and a note recording
the visibility gap. Future Claude Code versions that surface sidechain tool
calls would let the parser populate subagent reads via the `isSidechain` branch
already present in the code.

## Next Measurements

Before changing #103 behavior, collect at least:

- Post-`/clear` Claude protocol-read rows using an interactive trace and the
  parser above.
- Repeated Claude and Codex runs across task types to separate stable behavior
  from prompt-induced compliance.
- Codex behavioral recall rows with same-turn tool reads ruled out or recorded
  as exploration when a reliable tool-disabled Codex probe is available.
- Claude token-delta rows with per-file import ablation by commenting import
  lines rather than deleting files.
- Subagent measurement remains scoped to "which Agent invocations occurred and
  with which `subagent_type`" until a method to inspect subagent tool calls or
  initial prompts becomes available.
